Variants and editions

diagon ships one shared profile and toolchain across multiple deployment targets. Each variant reuses the same validation policy, service contracts, and diagonctl — only the packaging and boot mechanism differ.

At a glance

Comparison of diagon variants
Variant Platform Delivery Best for
Debian baseline Debian 12 (bookworm) simple-cdd installer ISO + systemd units The primary, CI-proven single-host deployment
OpenBSD variant OpenBSD Bootable raw disk image via autoinstall(8) Operators who prefer OpenBSD's base system
Compose fallback Debian Pinned-image compose.yaml + systemd unit Interim delivery when full packaging is unavailable

Debian baseline

The default and most thoroughly validated variant. It targets Debian 12 (bookworm), pins packaging dependencies through .github/integration-matrix.json, and produces an installer ISO via simple-cdd.

Differences from other variants

  • Uses the Debian preseed/package profile (myprofile.preseed, myprofile.packages).
  • Installs systemd units: diagon-i2pd.service, diagon-paywall.service, diagon-store.service.
  • Full eight-stage CI pipeline including Debian packaging verification.

When to choose it

Choose the Debian baseline for production or evaluation on Debian hosts — it is the reference deployment and receives the most complete CI coverage.

OpenBSD variant

The OpenBSD variant runs a separate CI pipeline and emits a directly bootable/installable raw disk image via OpenBSD autoinstall(8) under QEMU. It reuses the shared profile, validation policy, service contracts, bootstrap profile, and diagonctl tooling.

Differences from Debian

  • Driven by a shared OpenBSD config set: install.conf (autoinstall response, the preseed analog), install.site (a pkg_add hook), and pkg.list.
  • Uses a dedicated build matrix (.github/openbsd-matrix.json) with fields for OpenBSD version, architecture, and mirror.
  • Delivered as a raw .img disk image rather than an installer ISO.

When to choose it

Choose OpenBSD when you prefer its base system and packaging model, or need a directly bootable image. Configuration intent mirrors the Debian preseed/package set, so the marketplace topology is identical.

Compose/service fallback

An interim deliverable for Debian hosts if full packaging slips. It is CI-validated and comes with documented manual install steps.

What it includes

  • A pinned-image compose.yaml for i2pd, paywall, and store with dependency-aware startup ordering.
  • A diagon-compose.service systemd unit for host-level lifecycle control of the compose stack.
  • A secrets-aware environment template and a manual install guide.

When to choose it

Choose the compose fallback for a quick, reproducible bring-up on Debian while the full installer package is being finalized. Generate it with --emit-debian-compose-bundle-file (see the Installation guide).

Shared configuration across variants

Whichever variant you pick, the service topology, health endpoints, startup order, and i2pd tunnel mappings are validated by the same diagonctl contracts described on the Configuration page.

Hardware recommendations

All variants target a single host with local-only services. For any variant, ensure the host can run i2pd plus the marketplace services concurrently and supports the network configuration needed for live i2pd tunnels. See the Installation requirements for the baseline package and toolchain list.
