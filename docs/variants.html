Same stack, three ways to ship it

Variants

diagon ships one shared profile and toolchain across deployment targets. Each variant reuses the same validation policy, service contract, and diagonctl — only the packaging and boot mechanism differ.

01At a glance

02Debian baseline

The default and most complete variant. It targets Debian 12 (bookworm), pins packaging dependencies through the integration matrix, and produces an installer ISO via simple-cdd.

Differences

  • Uses the Debian preseed/package profile (myprofile.preseed, myprofile.packages).
  • Installs systemd units: diagon-i2pd, diagon-paywall, diagon-store.

Choose it when

You are deploying to a Debian host and want the reference experience.

03OpenBSD variant

Emits a directly bootable/installable raw disk image via OpenBSD autoinstall(8), reusing the shared profile, policy, contract, bootstrap profile, and diagonctl.

Differences

  • Driven by install.conf (autoinstall response — the preseed analog), install.site (a pkg_add hook), and pkg.list.
  • Uses a dedicated build matrix with OpenBSD version, architecture, and mirror.
  • Delivered as a raw .img disk image rather than an installer ISO.

Choose it when

You prefer the OpenBSD base system or need a directly bootable image. The marketplace topology is identical to Debian.

04Compose fallback

An interim deliverable for Debian if full packaging slips, with documented manual install steps.

Includes

  • A pinned-image compose.yaml for i2pd, paywall, and store with dependency-aware ordering.
  • A diagon-compose.service systemd unit for host-level lifecycle control.
  • A secrets-aware environment template and a manual install guide.

Choose it when

You want a quick, reproducible bring-up on Debian while the installer package is finalized. Generate it with --emit-debian-compose-bundle-file — see Reports.

Shared

Whichever variant you pick, the topology, health endpoints, startup order, and i2pd tunnel mappings are validated by the same diagonctl contracts described in Configure.
