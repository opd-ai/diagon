Configuration

diagon generates every component config from a single deployment profile. Secrets are injected from the environment or secure file paths — never hardcoded. This page documents the key files, options, and tuning knobs.

Configuration file locations

diagon configuration files
FilePurpose
profiles/myprofile.packagesPackage list for the installer profile.
profiles/myprofile.preseedPreseed answers (for example passwd/root-login, time/zone).
profiles/validation-policy.jsonPolicy contract defining required packages and preseed keys.
profiles/service-contract.jsonService topology: listeners, health endpoints, ordering, and i2pd tunnels.
profiles/local-single-host-bootstrap.jsonPhase 1 bootstrap profile with deterministic startup and local defaults.
.github/integration-matrix.jsonFrozen Debian baseline, dependencies, and component versions.
profiles/openbsd/OpenBSD variant config set (install.conf, install.site, pkg.list).

Policy file format

The policy contract defines required package and preseed sets per environment. *non-empty* is a sentinel meaning the key must exist and hold a non-empty value:

{
  "required_packages": ["curl", "i2pd", "openssh-server"],
  "required_preseed": {
    "passwd/root-login": "false",
    "time/zone": "*non-empty*"
  }
}

Service contract

The service contract validator enforces the marketplace topology. Key rules:

  • Required service definitions for i2pd, store, and paywall.
  • Local-only listener and health endpoint hosts: localhost, 127.0.0.1, or ::1.
  • Startup ordering: i2pd must start before store and paywall.
  • Dependency integrity with cycle detection.
  • Store → Paywall API link endpoint port compatibility.
  • i2pd tunnel type validation: client, http, http-proxy, socks, or server.
  • Local-only tunnel listener/target addresses with target-service port compatibility.
  • Required tunnel mappings for both store and paywall.

Bootstrap profile defaults

The bootstrap profile validator enforces Phase 1 single-host defaults:

  • Deterministic startup order: i2pd, paywall, store, diagonctl.
  • Local-only bind and health endpoints for all bootstrap components.
  • Default local Store → Paywall endpoint wiring and stubbed Paywall wallet mode.
  • Absolute, Debian-oriented config and secret file paths where file-backed secrets are used.
  • Expected i2pd tunnel names for the single-host bootstrap profile.

Secrets handling

Never hardcode secrets

Secrets are injected from environment variables or secure file paths. The only manual operator input beyond the profile defaults should be secrets — for example, a Monero wallet RPC endpoint.

For production wallet readiness, generate the wallet validation checklist, which captures secrets preflight, Monero wallet RPC probes, and paywall settlement verification commands:

go run ./cmd/diagonctl \
  --profile-dir profiles \
  --profile-name myprofile \
  --policy-file profiles/validation-policy.json \
  --bootstrap-profile-file profiles/local-single-host-bootstrap.json \
  --service-contract-file profiles/service-contract.json \
  --integration-matrix-file .github/integration-matrix.json \
  --integration-environment debian-12 \
  --emit-wallet-validation-checklist-file /tmp/diagon-wallet-validation-checklist.md \
  --format json

Generated config injection

Diagon can emit a config injection bundle wiring Store, Paywall, and i2pd paths and ports:

go run ./cmd/diagonctl \
  --profile-dir profiles \
  --profile-name myprofile \
  --policy-file profiles/validation-policy.json \
  --bootstrap-profile-file profiles/local-single-host-bootstrap.json \
  --service-contract-file profiles/service-contract.json \
  --emit-config-injection-file /tmp/diagon-config-injection.json \
  --format json

Use --emit-config-injection-file - to write the bundle to stdout.

Live probe options

When --probe-live is enabled, diagonctl performs runtime checks with capped retry backoff so transient startup races do not require a fixed high-frequency poll rate:

Live probe flags
FlagEffect
--probe-liveEnable runtime listener, health, tunnel, and sequencing checks.
--probe-timeoutOverall readiness timeout (for example 45s).
--probe-intervalBase retry interval between readiness attempts (for example 750ms).

Runtime probe output includes an aggregated component health view (aggregated_health) in JSON mode, marked failed when any component readiness probe fails.

Performance tuning

  • Backoff pacing. Tune --probe-interval up on slower hosts to reduce probe churn; the capped exponential backoff prevents busy polling.
  • Timeouts. Increase --probe-timeout when i2pd tunnel establishment is slow on first boot.
  • Startup ordering. Keep the frozen order (i2pd first) to avoid readiness races between dependent services.

Troubleshooting configuration issues

  • Validation fails on wildcard packages. These are warnings by default; running with --strict promotes them to errors.
  • Endpoint incompatibility warning. Confirm the Store → Paywall API link ports match in service-contract.json.
  • Non-local host rejected. Listeners and health endpoints must use localhost, 127.0.0.1, or ::1.
  • Readiness timeout. A dependent service became ready before its dependency, or a service never became ready — check startup order and increase the timeout.
