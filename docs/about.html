About diagon

diagon exists to make a Monero marketplace stack easy to deploy and operate on a single Debian host, with I2P connectivity built in and a CI path that proves it works.

Mission

The goal is to deliver a plug-and-play Monero marketplace stack for Debian that combines Diagon, Store, and Paywall over a locally running i2pd service — with a minimal CI path that proves the system can build, boot, and pass core integration checks. The project favors deterministic defaults, a single deployment profile, and an operator runbook so that non-expert users can provision and run the full stack.

Architecture

All components run as local services on one host:

  • Diagon — system orchestrator and operator entrypoint; manages startup order, health checks, and configuration wiring.
  • Store — the marketplace frontend/backend component.
  • Paywall — Monero payment gating and settlement logic.
  • i2pd — a local router daemon that starts first and publishes the I2P tunnels the services depend on.

Component configs are generated from one deployment profile, secrets are injected from the environment or secure file paths, and health/readiness probes are exposed per component for CI and operations.

Project status

diagon's roadmap works through alignment and contract freeze, local single-host bootstrap, Diagon integration wiring, a Debian service and packaging baseline, and release-candidate readiness — backed by an eight-stage CI validation plan. See the roadmap for the current checklist and progress log.

Contributing

Contributions are welcome. To get started:

  • Read the roadmap to understand scope and current phase.
  • Run go test ./... and validate profiles with diagonctl before opening a pull request.
  • Keep services local-only and secrets externalized, matching the existing contracts.
  • Open an issue to discuss significant changes first.
