General

What is diagon?

diagon is a plug-and-play Monero marketplace stack for Debian. It combines the Diagon orchestrator, Store, and Paywall over a locally running i2pd router, with a minimal CI path that proves the system can build, boot, and pass core integration checks.

What components make up the stack?

Four components: Diagon (orchestrator and operator entrypoint), Store (marketplace frontend/backend), Paywall (Monero payment gating and settlement), and i2pd (a local I2P router daemon providing connectivity).

Is this production-ready?

The roadmap targets a baseline single-host deployment with CI-proven build, boot, integration, contract, smoke, and packaging checks. Production hardening beyond baseline security controls is explicitly out of scope for the current roadmap.

What deployment model does diagon assume?

A single host running all components as local services. i2pd starts first and publishes local tunnel endpoints; Store and Paywall bind to local interfaces and route I2P traffic via i2pd tunnels.

Installation

Which operating systems are supported?

Debian 12 (bookworm) is the baseline. An OpenBSD variant is also available, delivered as a bootable raw disk image built via OpenBSD autoinstall(8).

What packages are required?

The baseline requires curl, openssh-server, and i2pd. The validation policy enforces these as required packages.

How do I install on Debian?

Install the prerequisites with apt, clone the repository, validate the profile with diagonctl, and generate the Debian package baseline. See the Installation guide for the full procedure.

Is there an option if the full installer package isn't ready?

Yes. A CI-validated compose/service fallback bundle for Debian is available with documented manual install steps. Generate it with --emit-debian-compose-bundle-file.

How do I verify a successful installation?

Run diagonctl with --probe-live. A healthy install shows services enabled and started in order, config files present, local health endpoints returning 2xx, i2pd tunnel listeners reachable, and an aggregated health view reporting all components ready.

Configuration

Where do configuration files live?

Under profiles/ — including the package list, preseed answers, validation policy, service contract, and the single-host bootstrap profile. See the Configuration reference.

How are secrets handled?

Secrets are injected from environment variables or secure file paths and are never hardcoded. The only manual operator input beyond profile defaults should be secrets, such as a Monero wallet RPC endpoint.

What is the policy file format?

A JSON contract with required_packages and required_preseed. The sentinel *non-empty* means a preseed key must exist and hold a non-empty value.

What startup order does diagon enforce?

Deterministic order: i2pd, then paywall, then store, then the diagonctl orchestration check. i2pd must always start before Store and Paywall.

What does strict mode do?

Strict mode (--strict) promotes warnings — such as wildcard package usage — into errors, which is useful for CI-grade validation.

Why must endpoints be local-only?

Listeners and health endpoints must use localhost, 127.0.0.1, or ::1. External reachability is provided exclusively through i2pd tunnels, keeping services off public interfaces.

i2pd & Monero

Which i2pd tunnel types are supported?

The service contract validates these tunnel types: client, http, http-proxy, socks, and server. Both Store and Paywall require tunnel mappings.

Do I need a Monero wallet to test?

No. Paywall supports a stubbed wallet mode for CI and local bootstrap. For production, use the wallet validation checklist to verify Monero wallet RPC readiness and paywall settlement.

How do I validate a production Monero wallet?

Generate the production wallet validation checklist with --emit-wallet-validation-checklist-file. It captures secrets preflight, Monero wallet RPC probes, and paywall settlement verification commands.

How does diagon handle i2pd startup races?

Runtime readiness checks use capped exponential backoff between attempts, and CI includes a readiness-timeout regression that fails if a required service never becomes ready. Tune --probe-timeout and --probe-interval for slower hosts.

Variants

What is the difference between the Debian and OpenBSD variants?

Both reuse the same profile, policy, contracts, and diagonctl tooling. Debian ships an installer ISO plus systemd units; OpenBSD runs a separate CI pipeline and produces a bootable raw disk image via autoinstall(8). See Variants.

When should I use the compose fallback instead of the full package?

Use the compose fallback for a quick, reproducible bring-up on Debian while the full installer package is being finalized. It is CI-validated and includes a manual install guide.

CI & contributing

What does the CI pipeline check?

An eight-stage pipeline: static checks, artifact builds, unit tests, integration bootstrap and readiness probes, contract tests, end-to-end smoke, Debian packaging verification, and checksum/release bundle publication. Merge is blocked unless Stages 1–7 pass; release is blocked unless Stage 8 passes.

How do I run the tests locally?

Run go test ./... from the repository root.

Where is the integration matrix defined?

In .github/integration-matrix.json, which pins the Debian baseline, packaging dependencies, upstream build inputs, and service-contract fixtures used in CI.
