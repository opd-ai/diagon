01General

What is diagon?

diagon is a plug-and-play Monero marketplace stack for Debian. It combines the Diagon orchestrator, Store, and Paywall over a locally running i2pd router, deployable from a single profile.

What components make up the stack?

Diagon (orchestrator), Store (marketplace), Paywall (Monero payment gating and settlement), and i2pd (local I2P router).

What deployment model does diagon assume?

A single host running all components as local services. i2pd starts first and publishes local tunnel endpoints; Store and Paywall bind to local interfaces and route I2P traffic via i2pd.

Is this production-ready?

It targets a baseline single-host deployment. Production hardening beyond baseline security controls is out of scope for the current roadmap; use the production wallet checklist before going live.

Why I2P instead of a public interface?

Services bind only to local interfaces; all external reachability travels through i2pd I2P tunnels. Nothing is exposed on a public network interface by default.

02Installation

Which operating systems are supported?

Debian 12 (bookworm) is the baseline. An OpenBSD variant is delivered as a bootable raw disk image built via autoinstall(8).

What packages are required?

curl, openssh-server, and i2pd. The validation policy enforces these as required packages.

How do I install on Debian?

Install the prerequisites, clone the repository, validate the profile, and generate the packaging plan. See the Install guide.

Is there an option if the full installer package isn't ready?

Yes — a compose/service fallback bundle for Debian with a manual install guide. Generate it with --emit-debian-compose-bundle-file.

How do I verify a successful installation?

Run diagonctl with --probe-live. A healthy install shows services started in order, config files present, health endpoints returning 2xx, tunnel listeners reachable, and aggregated health ready.

03Configuration

Where do configuration files live?

Under profiles/ — package list, preseed answers, validation policy, service contract, and the bootstrap profile. See the Configure page.

How are secrets handled?

Injected from environment variables or secure file paths, never hardcoded. The only manual operator input beyond profile defaults should be secrets, such as a Monero wallet RPC endpoint.

What is the policy file format?

JSON with required_packages and required_preseed. The sentinel *non-empty* means a preseed key must exist and be non-empty.

What startup order does diagon enforce?

i2pd, then paywall, then store, then the diagonctl check. i2pd must always start before Store and Paywall.

What does strict mode do?

--strict promotes warnings — such as wildcard package usage — into errors.

Why must endpoints be local-only?

Listeners and health endpoints must use localhost, 127.0.0.1, or ::1. External reachability is provided only through i2pd tunnels.

04i2pd & Monero

Which i2pd tunnel types are supported?

client, http, http-proxy, socks, and server. Both Store and Paywall require tunnel mappings.

Do I need a Monero wallet to test?

No. Paywall supports a stubbed wallet mode for local bring-up. For production, use the wallet validation checklist to verify Monero wallet RPC readiness and settlement.

How do I validate a production Monero wallet?

Generate the checklist with --emit-wallet-validation-checklist-file. It covers secrets preflight, Monero RPC probes, and settlement verification. See Reports.

How does diagon handle i2pd startup races?

Runtime readiness checks use capped exponential backoff. Tune --probe-timeout and --probe-interval for slower hosts.

05Variants

What is the difference between the Debian and OpenBSD variants?

Both reuse the same profile, policy, contract, and tooling. Debian ships an installer ISO plus systemd units; OpenBSD produces a bootable raw disk image via autoinstall(8). See Variants.

When should I use the compose fallback?

For a quick, reproducible bring-up on Debian while the full installer package is being finalized. It includes a manual install guide.

06Reports

What reports can diagon generate?

An operator runbook, bootstrap quickstart, wallet checklist, config-injection bundle, Debian package baseline, dependency manifest, compose fallback, release smoke plan, release baseline, and a Definition-of-Done report. Browse them on the Reports page.

How do I generate the reports myself?

Each is produced by a --emit-* flag on diagonctl. The Regenerate all section has a single command that produces the full set.

Can a report be printed to stdout?

Yes — pass - as the emit file path for any --emit-* flag to write to stdout instead of a file.
